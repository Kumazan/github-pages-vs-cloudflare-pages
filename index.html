    Beginner Guide 2026

    GitHub Pages 與 Cloudflare Pages
    差在哪裡？

    給第一次做公開網站、正在學 Vibe Coding 的新手。這頁會用簡單中文講清楚兩者差別，也順便說明 CI/CD 為什麼會讓做網站變得輕鬆很多。

    最簡單上線

    GitHub Pages

    適合純 HTML、作品集、筆記、說明頁。設定簡單，跟 GitHub repo 很貼近。

    更完整的部署流程

    Cloudflare Pages

    適合想要自動部署、預覽連結、全球 CDN、甚至未來加一點後端功能的人。

    一句話結論

    不是誰比較高級

    而是現在要的是「快速公開」還是「好迭代、好擴充、好協作」。

    · · ·
    Chapter 01

    先講人話，兩者到底是什麼？

    GitHub Pages 可以把靜態網站直接放到網路上。只要 repo 裡有 index.html，設定好 Pages，就能變成公開網址。

    Cloudflare Pages 也能做同樣的事，但它更像一條完整的網站發佈流水線。只要把修改 push 到 GitHub，它就能自動建置、產生預覽網址、再部署到正式站。

    簡單比喻：
    GitHub Pages 比較像「把做好的海報貼到牆上」。
    Cloudflare Pages 比較像「有一條會自動印刷、校稿、預覽、上牆的產線」。

    GitHub Pages

    強項是簡單

    如果只是想把一頁作品集、學習筆記、活動介紹頁放上網，它很快就能完成任務。

    Cloudflare Pages

    強項是流程

    如果會持續修改、想有預覽站、希望部署自動化，它通常比較順手。

    · · ·
    Chapter 02

    CI/CD 到底是什麼？

    很多人第一次看到 CI/CD 會覺得很工程師，但其實它的核心概念很生活化：

    修改檔案
    →
    Push 到 Git
    →
    系統自動建置
    →
    自動上線

    CI 可以理解成「每次內容更新，系統都自動幫忙檢查和組裝一次」。
    CD 可以理解成「檢查沒問題後，系統自動發佈」。

    用新手能懂的說法：CI/CD 就是「不用每次手動上傳檔案」，而是改完內容後，平台自己把網站更新好。

    那 GitHub Pages 沒有 CI/CD 嗎？

    不是。GitHub Pages 也可以有，尤其搭配 GitHub Actions 時一樣能做到自動部署。只是它的預設體驗比較偏向「把靜態頁面放上去」，而不是把預覽、分支部署、環境設定這些都做得很完整。

    為什麼這對 Vibe Coding 很重要？

    因為網站內容很可能會一直改。今天改標題、明天換配色、後天把 AI 產生的版面再調一下。有 CI/CD，每次 push 後都能很快看到新版本，不用反覆手動上傳。

    · · ·
    Chapter 03

    GitHub Pages 適合什麼情境？

    如果網站是 單頁 HTML、簡單 CSS、少量 JavaScript，那 GitHub Pages 其實很好用，而且很夠用。

    上手快

    如果已經有 GitHub repo，幾乎不用多學一套新東西。

    適合靜態內容

    作品集、課程介紹、教學筆記、活動頁、產品 landing page 都很適合。

    公開專案很自然

    網站內容和原始碼就在同一個公開 repo，分享與維護都方便。

    它的限制通常出現在這裡

    • 如果希望 每個 branch 都有預覽網址，GitHub Pages 預設沒有那麼直覺。
    • 如果想加一點後端邏輯，例如表單處理、API、身份驗證，GitHub Pages 就比較不是它的主場。
    • 如果之後常常改站、常常 review 版面，流程感會比較弱。

    一句判斷：如果只想「把一個完成版的靜態頁面放上網」，GitHub Pages 是很合理的選擇。

    Chapter 04

    Cloudflare Pages 多了哪些好用功能？

    Cloudflare Pages 比較像是「從簡單靜態頁出發，但保留未來升級的空間」。

    01

    Git 連動自動部署

    只要 push 到 GitHub，它就能自動重新建站，不用手動上傳檔案。

    02

    Preview 預覽連結

    每個 branch 或 PR 都能有一個預覽網址，很適合先給妹妹或朋友看再決定要不要正式上線。

    03

    全球 CDN 與 SSL

    Cloudflare 本來就很強在全球加速與 HTTPS，網站在世界各地通常都能很快打開。

    04

    自訂網域更順

    如果本來就用 Cloudflare 管 DNS，接自訂網域會很直覺。

    05

    環境變數與建置設定

    未來如果不只是一個 HTML，而是有 React、Astro、Next.js 之類的前端框架，就比較容易長大。

    06

    可延伸到 Functions

    如果哪天想加表單、簡單 API、驗證邏輯，Cloudflare 這條路比較自然。

    對非工程師最有感的一點通常是：
    Cloudflare Pages 會讓「改一點點內容 → 立刻拿到一個可分享的預覽網址」這件事變得很舒服。

    Chapter 05

    GitHub Pages 與 Cloudflare Pages 該怎麼選？

    選 GitHub Pages

    如果目標是最短路徑

    • 只有單頁 HTML 或簡單靜態站
    • 想今天就上線
    • 網站更新頻率不高
    • 不太需要 preview workflow
    選 Cloudflare Pages

    如果目標是更好的迭代流程

    • 會持續調整內容與版面
    • 想每次修改都自動部署
    • 想先分享預覽再正式發佈
    • 未來可能加表單、API 或框架

    還有一個很實際的差異：權限管理

    對很多人來說，真正關鍵的不只是部署快不快，而是 誰能改正式站、誰只能先看預覽、誰能碰設定和網域。

    GitHub Pages

    權限通常跟 repo 綁得比較緊

    誰能 push 到發佈分支，誰通常就能直接影響正式站。也就是說，程式碼權限和上線權限常常是綁在一起的。

    Cloudflare Pages

    比較容易把看預覽、改設定、正式部署拆開

    如果 repo 連到 Cloudflare Pages，能 push 到 production branch 的人一樣會影響正式站，但它多了 preview 連結、專案設定、網域與環境變數 這些管理層，協作時比較容易切清楚。

    • GitHub Pages 比較像：能改 repo 的人，多半也就是能動到網站的人。
    • Cloudflare Pages 比較像：可以先把 preview 網址給別人看，不一定要先給對方 repo 寫入權限或 production 權限。
    • 如果未來會碰 API key、表單、環境變數，Cloudflare Pages 通常也更適合做權限切分。

    最實際的建議

    如果只是這次要做一頁公開說明頁，GitHub Pages 完全可以先上。
    如果這頁之後會常改，或想順便學一套比較像現代前端工作流的部署方式，那就值得直接碰 Cloudflare Pages，尤其是在開始在意 preview、production 與權限管理之後。

    Chapter 06

    給正在學 Vibe Coding 的人，一個很實用的工作流

    對最近在學 Vibe Coding 的初學者來說，Cloudflare Pages 很適合當練習場，因為整個回饋循環很短。

    Step 1

    先叫 AI 生一版網站

    像這頁一樣，先做出可以看的單頁版，不追求一次完美。

    Step 2

    把修改 commit 到 GitHub

    每次改動都留痕跡，這樣比較知道自己改了什麼。

    Step 3

    讓平台自動部署

    不用再手動上傳檔案，網站會自己更新。

    Step 4

    先看預覽，再決定上線

    這一步很像在跟 AI 反覆對稿，特別適合設計與內容類網站。

    重點不是「用哪個平台比較潮」，而是有沒有建立一個能快速修改、快速看到結果、快速修正的迭代流程。

    · · ·
    Chapter 07

    如果目前就在 GitHub Pages，上 Cloudflare Pages 難嗎？

    如果目前的網站就是這種單頁 HTML，其實通常不難。因為靜態網站本來就很容易搬。

    1. 保留原本的 GitHub repo。
    2. 在 Cloudflare Pages 連接這個 repo。
    3. 如果只是純 HTML，通常幾乎不用特別 build 設定。
    4. 之後每次 push，Cloudflare 就會自動部署。
    • 不一定要二選一。很多人一開始在 GitHub Pages 練手感，之後想升級流程再接到 Cloudflare Pages。
    • 如果已經在用 Cloudflare 管網域，接自訂網址通常會更順。
    • 如果網站永遠都只是小型靜態頁，留在 GitHub Pages 也完全沒問題。
    · · ·
    Chapter 08

    最後濃縮成一句建議

    想先快速公開一頁網站，GitHub Pages 就夠了。

    想順便學現代網站部署流程，Cloudflare Pages 更值得碰。

    比較務實的建議是：
    這次這種單頁介紹站，可以先用 GitHub Pages 上線。接著再把同一個 repo 接到 Cloudflare Pages，順便學 preview deploy 和 CI/CD。這樣最容易看懂兩者的差別，也最不抽象。

    對新手來說，最好的學法不是一直看名詞，而是做一個真的能打開的頁面，然後親手體驗：

    • 我改了什麼？
    • 我 push 之後發生什麼？
    • 網站怎麼自動更新？
    • 預覽站和正式站差在哪？
    • 誰有權限影響正式站？

    搞懂這四件事，CI/CD 就不再只是名詞，而是實際會用到的 workflow。
