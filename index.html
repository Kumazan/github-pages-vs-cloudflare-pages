    Beginner Guide 2026

    GitHub Pages 與 Cloudflare Pages
    差在哪裡？

    給第一次做公開網站、正在學 Vibe Coding 的新手。這頁會用簡單中文講清楚兩者差別，也順便說明 CI/CD 為什麼會讓做網站變得輕鬆很多。

    最簡單上線

    GitHub Pages

    適合純 HTML、作品集、筆記、說明頁。設定簡單，跟 GitHub repo 很貼近。

    更完整的部署流程

    Cloudflare Pages

    適合想要自動部署、預覽連結、全球 CDN、甚至未來加一點後端功能的人。

    一句話結論

    重點在需求差異

    差異在於是否需要預覽部署、權限切分、可見對象控制，以及後續擴充。

    · · ·
    Chapter 01

    先講人話，兩者到底是什麼？

    GitHub Pages 可以把靜態網站直接放到網路上。只要 repo 裡有 index.html，設定好 Pages，就能變成公開網址。

    Cloudflare Pages 也能做同樣的事，但它更像一條完整的網站發佈流水線。只要把修改 push 到 GitHub，它就能自動建置、產生預覽網址、再部署到正式站。

    簡單比喻：
    GitHub Pages 比較像「把做好的海報貼到牆上」。
    Cloudflare Pages 比較像「有一條會自動印刷、校稿、預覽、上牆的產線」。

    GitHub Pages

    強項是簡單

    如果只是想把一頁作品集、學習筆記、活動介紹頁放上網，它很快就能完成任務。

    Cloudflare Pages

    強項是流程

    如果會持續修改、需要預覽站、並希望每次 push 後自動部署，這一類需求可以直接放進流程裡。

    · · ·
    Chapter 02

    CI/CD 到底是什麼？

    很多人第一次看到 CI/CD 會覺得很工程師，但其實它的核心概念很生活化：

    修改檔案
    →
    Push 到 Git
    →
    系統自動建置
    →
    自動上線

    CI 可以理解成「每次內容更新，系統都自動幫忙檢查和組裝一次」。
    CD 可以理解成「檢查沒問題後，系統自動發佈」。

    用新手能懂的說法：CI/CD 就是「不用每次手動上傳檔案」，而是改完內容後，平台自己把網站更新好。

    那 GitHub Pages 沒有 CI/CD 嗎？

    不是。GitHub Pages 也可以有，尤其搭配 GitHub Actions 時一樣能做到自動部署。差別在於 GitHub Pages 本身不會為每個 branch 自動建立預覽部署，也不內建 Access policy 這類可見對象控制。

    為什麼這對 Vibe Coding 很重要？

    因為網站內容很可能會一直改。今天改標題、明天換配色、後天把 AI 產生的版面再調一下。有 CI/CD，每次 push 後都能很快看到新版本，不用反覆手動上傳。

    · · ·
    Chapter 03

    GitHub Pages 適合什麼情境？

    如果網站是 單頁 HTML、簡單 CSS、少量 JavaScript，那 GitHub Pages 其實很好用，而且很夠用。

    上手快

    如果已經有 GitHub repo，幾乎不用多學一套新東西。

    適合靜態內容

    作品集、課程介紹、教學筆記、活動頁、產品 landing page 都是常見用途。

    公開專案很自然

    網站內容和原始碼就在同一個公開 repo，分享與維護都方便。

    它的差異主要出現在這裡

    • GitHub Pages 不會為每個 branch 或 PR 自動提供獨立的預覽網址。
    • GitHub Pages 主要提供靜態網站託管。表單處理、API、身份驗證通常要另外搭配其他服務。
    • 如果有多人協作，能 push 到發佈分支的人，就能更新正式站，repo 寫入權限和正式站更新會直接連在一起。
    • 如果要限制誰看得到網站，通常要再搭配其他機制，GitHub Pages 本身不提供 Cloudflare Access 這類 policy。
    • 如果每次修改都要先 review，再決定是否上線，通常要另外安排測試站、分支流程或其他工具。

    一句判斷：如果需求是把完成版靜態頁面直接公開，GitHub Pages 就能完成這件事。

    · · ·
    Chapter 04

    Cloudflare Pages 多了哪些好用功能？

    Cloudflare Pages 提供從 Git 觸發的建置與部署流程，並把預覽部署、正式部署、網域設定與後續擴充放在同一套系統裡。

    01

    Git 連動自動部署

    只要 push 到 GitHub，它就能自動重新建站，不用手動上傳檔案。

    02

    Preview 預覽連結

    每個 branch 或 PR 都能有一個預覽網址，可以先分享草稿版本，再決定是否上正式站。

    03

    全球 CDN 與 SSL

    Cloudflare 會把內容放到其 CDN，並預設提供 HTTPS，公開站點會透過同一套網路邊緣節點送出內容。

    04

    自訂網域與 DNS 可一起管理

    如果網域本來就在 Cloudflare，可在同一個平台處理 DNS、憑證、Pages 與 Access 設定。

    05

    可用 policy 控制可見對象

    除了誰能改、誰能部署，也可以透過 Access policy 管理誰能看到 preview。若有需要，還能把 pages.dev 或 custom domain 放到 Cloudflare Access 後面。

    06

    可延伸到 Functions

    如果要加表單、簡單 API、驗證邏輯，可以直接往 Pages Functions 延伸。

    這裡的差異可以直接拆成兩件事：
    一是 Cloudflare Pages 會自動提供預覽部署，二是可透過 policy 決定誰能看到 preview 或受保護的站點。

    Chapter 05

    GitHub Pages 與 Cloudflare Pages 該怎麼選？

    選 GitHub Pages

    適合這些情況

    • 只有單頁 HTML 或簡單靜態站
    • 要先把網站公開上線
    • 網站更新頻率不高
    • 不需要 branch / PR 預覽網址
    • 協作人數少，正式站更新權限不需要另外拆分
    選 Cloudflare Pages

    適合這些情況

    • 會持續調整內容與版面
    • 每次修改都要自動部署
    • 要先分享預覽，再決定是否上正式站
    • 未來可能加表單、API 或框架
    • 需要區分 preview、production、設定權限與可見對象

    最實際的建議

    如果只是這次要做一頁公開說明頁，GitHub Pages 可以直接使用。
    如果這頁之後會常改，或要把預覽部署、policy 與可見對象管理一起納入流程，Cloudflare Pages 會多出這幾個機制。

    Chapter 06

    給正在學 Vibe Coding 的人，一個很實用的工作流

    對最近在學 Vibe Coding 的初學者來說，Cloudflare Pages 可以直接練習「修改、push、預覽、部署」這一整段流程。

    Step 1

    先叫 AI 生一版網站

    像這頁一樣，先做出可以看的單頁版，不追求一次完美。

    Step 2

    把修改 commit 到 GitHub

    每次改動都留痕跡，方便對照每次修改的內容。

    Step 3

    讓平台自動部署

    不用再手動上傳檔案，網站會自己更新。

    Step 4

    先看預覽，再決定上線

    這一步很像在跟 AI 反覆對稿，特別適合設計與內容類網站。

    重點不是平台名稱本身，而是是否建立了「修改、預覽、確認、部署」這條迭代流程。

    · · ·
    Chapter 07

    如果目前就在 GitHub Pages，上 Cloudflare Pages 難嗎？

    如果目前的網站就是這種單頁 HTML，其實通常不難。因為靜態網站本來就很容易搬。

    1. 保留原本的 GitHub repo。
    2. 在 Cloudflare Pages 連接這個 repo。
    3. 如果只是純 HTML，通常幾乎不用特別 build 設定。
    4. 之後每次 push，Cloudflare 就會自動部署。
    • 不一定要二選一。很多人一開始在 GitHub Pages 練手感，之後想升級流程再接到 Cloudflare Pages。
    • 如果已經在用 Cloudflare 管網域，接自訂網址通常會更順。
    • 如果網站永遠都只是小型靜態頁，留在 GitHub Pages 也完全沒問題。
    · · ·
    Chapter 08

    最後濃縮成一句建議

    想先快速公開一頁網站，GitHub Pages 就夠了。

    想同時學預覽部署、policy 與可見對象控制，Cloudflare Pages 會多出這些內容。

    實作順序可以這樣安排：
    先用 GitHub Pages 上線這份單頁介紹站，再把同一個 repo 接到 Cloudflare Pages，對照 preview deploy、CI/CD、policy 與可見對象控制。

    對新手來說，最好的學法不是一直看名詞，而是做一個真的能打開的頁面，然後親手體驗：

    • 我改了什麼？
    • 我 push 之後發生什麼？
    • 網站怎麼自動更新？
    • 預覽站和正式站差在哪？
    • 誰有權限影響正式站？
    • 誰能看到預覽站或正式站？

    搞懂這幾件事，CI/CD 就不再只是名詞，而是實際會用到的 workflow。
